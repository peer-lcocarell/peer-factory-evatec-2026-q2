## Findings

### HIGH Severity

**H1 — Preconditions use bullet list format instead of sentence format**

All 75 test cases use bullet list preconditions. The established project standard requires complete sentences in paragraph form, not bullet points.

- Found: `- User is signed in with permission to view Magnet System Types.`
- Required: `The user has signed into the application with the 'MagnetSystem_View' security right. The 'Magnet System Types' page is accessible.`

Affects: All test cases across all four CRs.

**H2 — Security rights are not cited with specific right names in single quotes**

Most test cases describe permissions generically ("User is signed in with permission to import") rather than naming the specific security right in single quotes.

- Found: `User is signed in with permissions to import and view Hardware Components.`
- Required: `The user has signed into the application with the 'BPS_Edit' and 'BPS_View' security rights.`

Specific right names must be confirmed with the development team and documented.

Affects: All test cases where a security right controls access.

**H3 — Prohibited language in Expected Outcome section**

Several test cases use the prohibited phrase "as expected" or vague language in their Expected Outcome summary sections.

Examples found:
- `Tester can view and interact with the Magnet System Types list as expected.` (290756)
- `The status and history update accordingly.` (290772)
- `Correct field behavior.` (290757)

Required: Specific, measurable state changes must be described (e.g., "The status column displays 'Retired' and a history record is visible in the 'Magnet System History' page.").

Affects: Approximately 20+ test cases.

**H4 — CR 286067 missing core lifecycle state machine coverage**

The highest-risk CR has the following lifecycle gaps:

Missing lifecycle stages for Magnet Systems:

- Checkout (check out a magnet to a technician/user)
- Install into tool position
- Uninstall from tool position
- Status transition validation (e.g., prevent invalid transitions such as Retired → Active)
- Promotion: Standard to Prototype
- Promotion: Prototype to Standard

Missing for Magnet System Types:

- Deactivation / retirement of a Magnet System Type
- Attempt to use a retired type in new Magnet System creation

Migration coverage (existing hardware components → Magnet Systems) is absent.

### MEDIUM Severity

**M1 — Title convention does not follow established qualifier prefix pattern**

Current format: `R10.2 Update for CR 286067 - Feature Name: Verify action (Happy Path)`

Established project format: `'Happy Path': Verify [action] in the '[Feature Name]' page`

The qualifier should prefix the title, and the feature area should be in single quotes. The parenthetical `(Happy Path)` at the end is not consistent with the corpus.

Affects: All 75 test cases.

**M2 — "Tester can..." phrasing in Expected Outcome inconsistent with corpus**

The corpus uses "The user..." throughout preconditions, steps, and expected results. The new test cases use "Tester can..." in the Expected Outcome section.

- Found: `Tester can retire an active Magnet System and the status and history update accordingly.`
- Required: `The user is redirected to the 'Magnet Systems' list. The status column shows 'Retired'. A history entry is displayed for the retirement event.`

Affects: All test cases with an Expected Outcome section.

**M3 — Expected Outcome section is non-standard**

The `## Expected Outcome` section appended after the steps table is not part of the ADO-compatible template defined in the project instructions. The publishing utility parses only the Preconditions section and the Steps table. This section will not be published to ADO and may cause confusion.

Recommendation: Remove the `## Expected Outcome` section or confirm it will be excluded from ADO publishing. If retained, it must contain measurable language only.

Affects: All 75 test cases.

**M4 — CR 285729 missing partial text search and post-modification search coverage**

CR 285729 (Search HW Components) has the following coverage gaps:

- Search by partial value (substring matching)
- Search results refresh after a hardware component is edited
- Search after a component is moved to a different module
- No-results scenario when valid criteria returns zero matches
- Performance validation when large number of records exists

Current coverage: search by Part Number, Serial Number, Description, date range, docked/undocked scope — 7 cases total. Core positive paths are covered. Negative and edge cases are thin.

**M5 — CR 286836 missing BOM import and partial import disabled validation**

CR 286836 (Article Import) has the following coverage gaps:

- BOM import via Excel (Load BOM from Excel — requirement from R10.0.7 precedent)
- Partial import being disabled (block importing subset of records)
- Work Order import when duplicate BPS# exists
- Import summary counts validation (added / updated / skipped)

**M6 — CR 284111 missing WBS edit on existing box and display validation**

CR 284111 (WBS Element) has the following coverage gaps:

- Edit WBS element on an existing box that was not originally created as Self-Service
- Display validation: WBS field visible and non-editable in View mode
- Search and filter by WBS Element value on the Boxes list
- Reporting impact: does WBS appear in exported box data?
