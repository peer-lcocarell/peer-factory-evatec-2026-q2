### CR 286836 – Article Import

**High-Level Summary**

This CR enhances the Hardware Component (Article) import process by introducing support for configurable import formats and improving compatibility with updated PLM export files. The change removes dependence on a single import structure, allowing users to import hardware component data from different file formats without requiring application changes or restarts.

In addition, the CR addresses issues with import processing, including handling status values correctly and preventing valid Hardware Components from being unintentionally deactivated when importing partial article lists or BOM-driven datasets.

### Business Value

* Supports changing PLM export formats without development changes.
* Provides a more flexible and maintainable import framework.
* Reduces manual effort required to prepare import files.
* Prevents accidental deactivation of valid Hardware Components.
* Improves data integrity and reliability of Hardware Component imports.

### Key Areas Impacted

* Hardware Component Import
* ERP Import functionality
* BOM Import functionality
* Work Order Import functionality
* Import format configuration and mapping
* Hardware Component maintenance

### QA Focus

* Validation of configurable import formats and mappings.
* Hardware Component imports using legacy and new file formats.
* Correct field mapping and status value handling.
* Verification that partial imports do not incorrectly deactivate existing components.
* Regression testing of shared import framework functionality across ERP, BOM, and Work Order imports.

### Risk Level

**Medium**

The change affects a shared import framework used by multiple business processes. Incorrect mappings or import logic could introduce data integrity issues, impact existing import workflows, or lead to unintended Hardware Component activation/deactivation behavior.

### Requirements Impacted

**CR 286836 - Ticket #34948 - Article Import**

* **75372**: Edit ERP Hardware Component (Hardware Components)
**Difference introduced by the change:**

New functionality was added for users with the **ERP\_Edit** security right:

* Users can now **select one or more hardware components and toggle the state of the "Deleted" column**.
* This capability was **not present in the previous version**.

Everything else remains unchanged:

* Users can modify imported ERP comments.
* Save stores changes.
* Cancel closes without saving.
* ERP\_Edit security right is required.
* Article # column provides a link to the Edit feature for authorized users.

**Impact:**

* Scope expanded from only editing ERP component comments to also managing the **Deleted status** of hardware components in bulk or individually.


* **264374**: Import Hardware Components from File (Hardware Components)


**Difference introduced by the change:**

The import behavior was expanded to update an additional object type.

**Old behavior**

* Any **target types** defined from an ERP hardware component would have their:
  * Name (English & German) updated
  * Status updated

**New behavior**

* Any **target types and magnet system types** defined from an ERP hardware component will have their:
  * Name (English & German) updated
  * Status updated

**Impact**

* Imports now propagate updates to **magnet system types** in addition to **target types**.
* This increases the scope of the hardware component import process and may require additional regression testing to verify magnet system type names and statuses are updated correctly during import.

No other changes were made to:

* File format mappings
* Product group filtering
* Import summary reporting
* Security requirements
* Custom file format rules and mappings.


* **272565**: Import Work Orders from File (Work Orders)
**Key differences introduced by the change:**

### 1. Import supports multiple file formats

**Old**

* Only a **Project File** could be imported.

**New**

* Work orders can be imported from **various file formats**.
* A **format selector** is available.
* PF defaults to the **"project file"** format.

### 2. Field mapping changes

**Old Mapping**

| PF Name          | File Column Name  |
| ---------------- | ----------------- |
| Article #        | Project           |
| Name             | Project Name      |
| Application Code | Project Type      |
| Status           | Processing Status |

**New Mapping**

| PF Name      | File Column Name  |
| ------------ | ----------------- |
| Project #    | Project           |
| Name         | Project Name      |
| Project Type | Project Type      |
| Status       | Processing Status |

Changes:

* **Article #** renamed to **Project #**
* **Application Code** renamed to **Project Type**
* Added note that Project Type was previously called Application Code

### 3. Custom file formats added

**New functionality**

* Users can create and edit custom import formats.
* Custom formats require:
  * Unique name (max 50 characters)
  * Mapping for **Project #** and **Name**
* Optional mappings:
  * Project Type
  * Status
  * Equipment #
  * Product Group
  * Product Type
  * Business Unit

**Old**

* No custom file format support described.

### 4. Import configuration expanded

**New**

* User selects both:
  * Import format
  * Import file

**Old**

* User only selected a project file.

### Unchanged

* Closed or Locked statuses are imported as **Closed**.
* Import summary shows counts of added, updated, and skipped work orders.
* Requires **ERP\_View** and **ERP\_Edit** security rights.

### Impact

This is a significant enhancement to the import feature. The functionality moves from a fixed project-file import to a configurable import framework with support for multiple file formats, custom mappings, and additional optional work order fields.
